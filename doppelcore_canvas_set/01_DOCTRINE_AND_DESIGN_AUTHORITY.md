# Canvas 01 — Doctrine and Design Authority

**Date and Time:** 2026-04-18 07:44:18

## 1. Design authority

The operator is the owner and final policy authority.

This means DoppelCore is not constrained by committee-style governance assumptions. Protocols, registries, and document systems exist to serve the operator's objective. If an older protocol shape blocks a more truthful code mirror, the protocol is the thing that changes.

## 2. Primary objective

The primary objective is:

> Build a system that is as close to a mirror of the code as possible.

Everything else is secondary.

## 3. Truth hierarchy

### Tier 0 — Reality
The repository at a specific path and revision.

### Tier 1 — Machine-readable mirror
DoppelCore records derived from bounded evidence.

### Tier 2 — Governance control
Registry decisions, remediation actions, verification outcomes, and operator approvals.

### Tier 3 — Human surfaces
Compiled `SYSTEM.md`, UI pages, posture summaries, diagrams, review packets.

Only Tier 0 and Tier 1 are truth-near. Tier 3 is a projection layer.

## 4. What DoppelCore is

DoppelCore is:

- a machine-readable truth kernel
- contract-bound
- evidence-bound
- posture-aware
- drift-aware
- deterministic where possible
- explicitly partial where proof is incomplete

## 5. What DoppelCore is not

DoppelCore is not:

- a freeform narrative generator
- a generic autonomous agent
- a replacement for Registry control logic
- a silent inference engine that upgrades guesses into facts
- a UI-first documentation gimmick

## 6. Governing doctrine

### Rule 1 — code first
Code and file evidence come before prose.

### Rule 2 — machine before narrative
Canonical record creation happens before human explanation.

### Rule 3 — explicit unknowns
Any incomplete proof must be recorded as unknown, partial, or unassessable.

### Rule 4 — posture is mandatory
Every major record or aggregate must carry posture, not just pass/fail language.

### Rule 5 — no beautiful lies
A polished render that outruns machine evidence is a defect.

## 7. Success condition

DoppelCore succeeds when the operator can ask:

- what exists
- what the system claims
- what evidence supports the claim
- what is missing
- where drift exists
- what is deterministic versus inferred

and get a precise answer without trusting a narrative alone.
